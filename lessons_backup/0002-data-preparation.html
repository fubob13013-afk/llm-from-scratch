第一步：分词 —— 把句子「切成块」

为什么要切？

计算机无法直接理解「我喜欢人工智能」这句话。它需要一个翻译器，把人类语言翻译成它能处理的整数序列。这个翻译器的第一步，就是把句子切成一个个最小处理单元——称为 Token（词元）。

三种切法，哪个更好？

方案示例（"人工智能很有趣"）优点致命缺陷
按词切 ["人工智能", "很", "有趣"] 语义完整 词表无限膨胀：新词永远在出现，"元宇宙"、"i人e人"……词表大到无法管理
按字切 ["人", "工", "智", "能", "很", "有", "趣"] 词表极小（几千个常用字搞定） 序列太长：一句话变几十个 token，模型处理慢，且丢失词语级别的语义
子词级（BPE） ["人工", "智能", "很", "有趣"] 词表可控 + 能处理新词 ✅ 当前所有主流 LLM 的选择
🔑 类比 按词切 = 每道菜都单独起一个名字（菜单无限长）
按字切 = 每道菜拆成食材颗粒（描述起来又臭又长）
BPE = 找到最常见的「食材组合」，比如"番茄炒蛋"记成一个单位，"青椒肉丝"记成一个单位——既不会菜单无限长，也不会描述得太碎。

核心算法：BPE（Byte Pair Encoding）

BPE 是 GPT-2/3/4、LLaMA、Qwen、DeepSeek 等几乎所有模型都在用的分词算法。它的思路极其简单：

从字符级开始，反复合并出现频率最高的相邻 token 对，直到词表达到预设大小。

手动走一遍 BPE

假设我们有这样一份迷你语料（只有 4 个词，每个词后加 </w> 表示词尾）：

low</w>    lower</w>    newest</w>    widest</w>

初始状态——拆成字符：

l o w </w>    l o w e r </w>    n e w e s t </w>    w i d e s t </w>

词表 = {l, o, w, e, r, n, s, t, i, d, </w>}，共 11 个。

现在开始合并。每一步找出现次数最多的相邻对：

第 1 轮：统计所有相邻对出现次数—— "lo"=2, "ow"=2, "w</w>"=1, "we"=2, "er"=2, …… 出现最多的有几组并列，选 "e" + "s" = "es"，出现 2 次（newest, widest），合并为新 token "es"。

语料变为：l o w </w> l o w e r </w> n e w es t </w> w i d es t </w>
词表 += "es"，共 12 个。
第 2 轮："es"+"t" = "est"，出现 2 次（newest, widest）。

语料变为：l o w </w> l o w e r </w> n e w est </w> w i d est </w>
词表 += "est"，共 13 个。
第 3 轮："l"+"o" = "lo"，出现 2 次（low, lower），合并。
词表 += "lo"，共 14 个。

继续重复……假设词表上限设为 16，做完后发现：

原始词："low"     → token: ["lo", "w</w>"]
原始词："lower"   → token: ["lo", "w", "er</w>"]
原始词："newest"  → token: ["n", "e", "w", "est</w>"]
原始词："widest"  → token: ["w", "i", "d", "est</w>"]

看到了吗？"low" 和 "lower" 共享了 "lo" 和 "w"；"newest" 和 "widest" 共享了 "est"。高频组合被合并成独立 token，低频组合保留在字符级。这就是 BPE 的魔力——用有限的词表，覆盖无限的词汇。

BPE 在中文里

中文 BPE 的初始单位通常是字节（UTF-8 编码）或字符。以「人工智能很有趣」为例，UTF-8 编码后每个汉字占 3 个字节，BPE 会在字节流上做合并。实际效果是：常见词组（"人工智能"、"学习"）会合并成一个 token；罕见组合保留为单个字。

一个经验数据：1 个中文 token ≈ 1~1.8 个汉字；1 个英文 token ≈ 0.75 个单词 ≈ 3~4 个字母。这就是为什么同样的内容，中文需要的 token 数通常比英文少。

第二步：Token → ID（词表映射）

BPE 跑完后，我们有了一个词表（Vocabulary）——每个 token 对应一个唯一的整数 ID。比如：

Token"lo""w""er""est""</w>""n""e""i""d"……
ID012345678……

把文本翻译成 ID，就是查表：

输入文本:  "lower"
分词结果:  ["lo", "w", "er</w>"]
Token ID:  [0, 1, 2]

这就是模型真正「看到」的东西——一串整数。你喂给 GPT 的每一个字，在进入模型之前都已经被转换成了这样的数字序列。

第三步：特殊 Token

除了普通词汇，词表里还有一些功能性 token：

[BOS] / <|startoftext|>
句子开头标记。告诉模型：「这是一段文本的起点」
[EOS] / <|endoftext|>
句子结尾标记。模型生成这个 token 时，意味着「我说完了，可以停了」
[PAD]
填充标记。一个 batch 里句子长短不一时，短句后面补 PAD 对齐
[UNK]
未知词标记。遇到词表里不存在的字符时用这个替代（现代 BPE 基本不会出现，因为 BPE 能从字符级重建任何词）
原始句子: "我喜欢学习人工智能"
处理后:   [BOS] 我 喜欢 学习 人工 智能 [EOS]
Token ID: [ 0,   15, 42,  38,  89,  91,  1 ]

第四步：滑动窗口 —— 造出训练样本

现在所有文本都变成了数字序列。但怎么把它变成「输入 → 正确答案」的训练对？

用的是滑动窗口——这是整个数据准备里最精妙的设计。

假设窗口大小 = 4（实际 GPT 训练窗口通常是 1024~8192），有一个 token 序列：

[BOS] 我 喜欢 学习 人工 智能 很 有趣 [EOS]
  0    1   2    3    4    5   6   7    8

滑动窗口从最左边开始，每次向右挪一格。「输入」是当前位置看到的内容，「目标」是下一个位置要预测的 token：

样本 #输入（模型看到的）目标（模型要猜的）
1[0, 1, 2, 3] → [BOS] 我 喜欢 学习[1, 2, 3, 4] → 我 喜欢 学习 人工
2[1, 2, 3, 4] → 我 喜欢 学习 人工[2, 3, 4, 5] → 喜欢 学习 人工 智能
3[2, 3, 4, 5] → 喜欢 学习 人工 智能[3, 4, 5, 6] → 学习 人工 智能 很
4[3, 4, 5, 6] → 学习 人工 智能 很[4, 5, 6, 7] → 人工 智能 很 有趣
5[4, 5, 6, 7] → 人工 智能 很 有趣[5, 6, 7, 8] → 智能 很 有趣 [EOS]
🔑 核心洞察 注意：输入和目标来自同一条序列，只是错开了一位（shifted by 1）。这是 LLM 训练的基石——模型永远在「看到前面，猜下一个」。目标序列就是输入序列往右移一格。

所以不需要人工标注！数据自己就是自己的标签。这就是「自监督学习」（self-supervised learning）——训练数据自己提供监督信号。

第五步：DataLoader —— 组装成批次

最后一步是工程上的：把几百万个训练样本打包成批次（batch），每次喂一小撮给模型。这涉及三个紧密配合的操作。

前置知识：CPU 和 GPU 的区别

🔑 核心类比 CPU = 一个数学天才，能心算微积分，但一次只能算一道题。
GPU = 1000 个小学生，每人只会做加减法，但同时做 1000 道题。

训练神经网络的运算恰好是「大量简单运算同时做」——矩阵乘法就是几万个加减法同时算。这种工作丢给 CPU 是折磨天才，丢给 GPU 是小学生狂欢。

① Batching（批处理）—— 为什么不能一个一个喂？

GPU 有几千个核心。如果一次只喂 1 个样本，几千个核心里只有几个在工作——利用率大约 3%。而且数据从内存搬到 GPU 的「搬运费」是固定的：搬 1 个样本和搬 64 个样本耗时差不多。

所以 batch_size=64 的意思是：每次搬 64 个样本一起算，搬运费只花一次，GPU 几千个核心也都有活干。

🔑 食堂托盘类比 食堂阿姨不会给你一碗一碗端——她一次端一个托盘（8~64 碗），走一趟就能喂饱一桌人。
batch_size = 托盘能装几碗。太小：跑断腿。太大：端不动（显存爆了）。

② Padding（填充）—— 句子长短不一怎么办？

GPU 做矩阵运算时，数据必须是矩形——每行长度必须相同。但真实句子天然长短不一：「天气真好」只有 4 个 token，「我喜欢学习人工智能」有 7 个 token。

Padding 的做法：同一批里，所有句子垫到和最长的那条一样长，短的在后面补 [PAD]。

那模型会不会把 [PAD] 当真内容学了？不会。训练时有一个叫 Attention Mask（注意力掩码） 的机制，明确告诉模型这些 PAD 位置是假的，计算时跳过它们。具体到 Attention 那课展开。

🔑 搬家打包类比 不同大小的家具装进统一尺寸纸箱——小家具的箱子里塞气泡膜填空。
家具 = 句子内容，纸箱 = 统一长度，气泡膜 = [PAD]（到目的地就扔）。

③ Shuffling（打乱）—— 为什么要洗牌？

假设训练数据按类别排列：前 5000 条科技，中间 5000 条娱乐，后 5000 条体育。如果不打乱，模型连续看 5000 条科技后会发现：「反正这一片全是科技，我猜科技词汇就对了」——它在利用数据排列顺序作弊，而不是学习语言规律。

就像背单词卡片永远按字母顺序背，你会发现你能按顺序背出来，但单独抽一张就不认识——你记住的是「abandon 后面是 ability」，而不是真正学会了单词。

Shuffling = 每个 epoch 开始前洗一次牌，让模型没法靠顺序偷懒。

⚠️ 注意顺序：Shuffle 必须在分组之前。如果先分组再打乱组的顺序，组内样本还是按原来的顺序相邻——作弊信号还在。

PyTorch 的 DataLoader 负责这整条流水线——你把数据集给它，它返回一个迭代器，每次 next() 吐出一批处理好的 (input, target) 对。
